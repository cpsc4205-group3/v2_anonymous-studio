┌─────────────────────────────────────────────────────────────────────────────────────┐
│                          KEY PERFORMANCE OPTIMIZATIONS                              │
└─────────────────────────────────────────────────────────────────────────────────────┘

1. Operator Config Caching (_OPS_CACHE)
   - Caches OperatorConfig objects per (operator, entities) key
   - Prevents rebuilding 17+ configs per cell in batch jobs
   - Result: ~2x speedup on large datasets (10k+ rows)

2. Denylist Pattern Caching (_DENYLIST_PATTERN_CACHE)
   - Compiles regex patterns once and reuses
   - Avoids repeated Pattern compilation overhead

3. Background Warmup Thread
   - Loads Presidio engines at import time (non-blocking)
   - Eliminates 1-2s first-call latency
   - User sees instant response on first analysis

4. Chunked Processing (Batch Jobs)
   - Processes large CSVs in 100-row chunks
   - Updates PROGRESS_REGISTRY after each chunk
   - Prevents memory exhaustion on million-row datasets

5. Lazy Engine Initialization
   - Presidio engines only created on first use
   - Fast app startup (imports take <1s)
   - Defers expensive spaCy model loading

│                              COMMON PITFALLS                                        │
└─────────────────────────────────────────────────────────────────────────────────────┘

❌ Blank spaCy Model → PERSON/LOCATION/ORG never detected
   ✅ Solution: python -m spacy download en_core_web_lg

❌ Thread Safety → Concurrent users can crash Presidio engines
   ✅ Solution: Use locks or per-user engine instances

❌ Large Text Blocking UI → Multi-megabyte text blocks freeze interface
   ✅ Solution: Use invoke_long_callback() for text >10KB

❌ Random Hash Salt → Same PII produces different hashes each run
   ✅ Solution: Pass explicit salt parameter for referential integrity
